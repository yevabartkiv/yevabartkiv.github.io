Understanding k-Book concept

k-Book has information about all recurring enterprise tasks and their step-by-step how-to accomplish guidance.

The base idea of Knowledge Book (k-Book) is that 90% of enterprise tasks are recurring. That means that someone have already done the same task before and all you need to be effective in your own task completion, is to get the how-to information. k-Book is Role-Tasks centric and gives you all enterprise domain knowledge needed to accomplish your Task. Choose your Task from the list and follow the instruction until the Task is done.

After the Task is done, you'll be able to use the very same method regarding your next task.

All enterprise tasks (equal to procedures) are divided to processes (Sales Tasks) and projects (t7001. Project Kickoff). Projects in turn are usually implemented within a process. Tasks are assigned to Roles. Some of the enterprise roles as well as their tasks may be outsourced (example: some marketing and sales tasks may be outsourced to partners). There are so called Application Tasks that end user can accomplish by using application features. These tasks are the part of all enterprise tasks or in case your enterprise doesn't use the application are not your enterprise tasks but your clients tasks.

One of the key benefit of using k-Book is accuracy of searching for information results. To get that accuracy and ease searching process all information inside k-Book is divided (structured) to separate information units. All information units such as tasks, projects, processes, products (solutions), product features, product modules, module components (source code, UI elements, UI wizards, events and actions) are supplied with ID. Each ID starts with letter and follows by up to 4 digit numbers.

  • t - task (the same as procedure). Task has four digit numbers, where first number stands for: 0--- application task; 1--- administrative task; 2--- product planning task; 3--- codding task; 4--- marketing task; 5--- sales task; 6---after sales support task; 7--- project specific task; 8--- partner specific task; 8--- management task. Example: t7001. Project Kickoff

  • r - role. Role has three digit numbers, where first number stands for: 0-- product (application) end user role; 1-- enterprise role ; 2-- outsourced enterprise role to partner. Example: r02. Technical Account Guarantor

  • s - solution, 01 - sequence number of solution (project). Example: s01. title_name

  • p - product, 01 - sequence number of product. Example: p01. Identity Manager (IdM)

  • f - product feature, 11 - sequence number of feature. Example: f01. Synchronization

  • m - application module, 07 - sequence number of application module. Example: m05. Access Management

  • c - component. Components have four digit numbers, where first number stands for: 0--- source code component; 1--- UI element, 2--- UI wizard. Example: c2002. Assign Role; c1007. Technical Account (c2002)

  • e - event. Event has four digit numbers, where first number stands for: 0--- application event; 1--- task event. Example: e0404. title_name

  • a - action. Action has four digit numbers, where first number stands for: 0--- action on application event; 1--- action on task event. Example: e0404. title_name

    There are several searching possibilities in k-Book:

  • using information units IDs

  • using contextual searching

  • using table of contents

    Subject matter experts (SME) and technical writer form the party which is responsible for k-Book development. You can find the information about the party in the Party section.
